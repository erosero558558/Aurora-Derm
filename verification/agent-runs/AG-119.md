# AG-119 - Turnero sala W: transparencia de fallback parcial en admin queue

## Objetivo
Evitar confusion operativa cuando `admin.html#queue` entra en modo fallback por `queue-state`, dejando explicito cuando la vista es parcial y no una lectura completa de la cola.

## Cambios implementados
- `src/apps/admin/modules/queue.js`
  - Nuevo contexto de fallback en UI (`fallbackContext`) para guardar:
    - `waitingCount`, `calledCount`
    - cantidad muestreada en tabla (`nextTickets + callingNow`)
    - bandera `partial` cuando backend reporta mas tickets de los mostrados.
  - La barra de resumen de triage ahora agrega indicador:
    - `fallback parcial X/Y` cuando aplica.
  - El panel "Siguientes turnos" agrega nota operativa:
    - `Mostrando primeros N de M en espera (fallback).`
  - Se limpia el contexto de fallback al volver a sincronizacion `live` o snapshot.
- `tests/admin-queue.spec.js`
  - Escenario reforzado de fallback:
    - `/data` sin `queue_tickets`
    - `queue-state` con `waitingCount` mayor al top de tickets
    - valida indicador `fallback parcial` y nota de muestra parcial.
- `docs/RUNBOOK_TURNERO_SALA.md`
  - Seccion de contingencia actualizada para dejar explicito que fallback puede ser muestra parcial.

## Validacion ejecutada
- `npx playwright test tests/queue-kiosk.spec.js tests/queue-display.spec.js tests/queue-integrated-flow.spec.js` -> `18 passed`.
- `npx playwright test tests/admin-queue.spec.js` -> verde tras ajuste de fallback.

## Resultado operativo
- Recepcion ve cuando la cola es completa vs muestra parcial durante degradacion.
- Se reduce falso diagnostico de "cola vacia" cuando backend reporta tickets pero el modo es fallback.
